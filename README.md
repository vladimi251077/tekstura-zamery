# Tekstura Замеры

Tekstura Замеры — рабочее PWA-приложение для выездных замерщиков лестниц Tekstura. Приложение помогает создать карточку замера, сохранить размеры, фото, SVG-схему, комментарии и передать подготовленные данные менеджеру и изготовителю/конструктору.

Проект уже используется в продакшене, поэтому любые изменения должны идти маленькими безопасными PR. Большой рефакторинг, переписывание архитектуры и массовые изменения рабочих файлов нужно делать только поэтапно.

## Для кого приложение

- **Замерщик** — создаёт новый замер на объекте, добавляет размеры, фото, комментарии и схему.
- **Менеджер / администратор** — смотрит список замеров, проверяет статусы, принимает, архивирует, восстанавливает и очищает корзину.
- **Изготовитель / конструктор** — открывает режим «Для изготовителя», чтобы видеть подготовленные размеры, схему, фото и производственные данные без лишнего интерфейса замерщика.

## Основные возможности

- создание замеров и карточек клиента;
- сохранение размеров проёма, маршей, площадки/забежных ступеней и комментариев;
- загрузка фото с камеры или из галереи;
- сохранение SVG-схемы лестницы;
- режим «Для изготовителя» через `production.html`;
- offline TEMP-черновики на телефоне;
- offline-фото в локальном хранилище телефона;
- синхронизация TEMP-черновиков и фото с Supabase после появления интернета;
- список замеров, фильтры, статусы, архив;
- корзина, восстановление и полное удаление;
- очистка Supabase через пароль полного удаления.

## Как запустить локально

Проект сейчас работает как статическое приложение без обязательного build-step.

```bash
# из корня проекта
python3 -m http.server 5178
```

На Windows можно использовать:

```powershell
py -m http.server 5178
```

После запуска открыть:

```text
http://127.0.0.1:5178/
```

Для проверки PWA/offline лучше открывать через `localhost` или `127.0.0.1`, потому что service worker работает только в secure context. `localhost` считается допустимым secure context для разработки.

## Главные файлы

- `index.html` — основной экран приложения, вход, список замеров, форма, фото, технические действия.
- `app.js` — основной runtime приложения: авторизация, роли, замеры, фото, offline TEMP-черновики, синхронизация, корзина.
- `offline-db.js` — IndexedDB-слой для локальных TEMP-черновиков, offline-фото и служебных метаданных.
- `service-worker.js` — PWA/offline shell, кэширование основных файлов приложения.
- `drawing-bridge.js` — мост между карточкой замера и SVG-конструктором/схемой.
- `production.html` / `production.js` — отдельный режим «Для изготовителя».
- `styles.css` — стили основного приложения.
- `manifest.webmanifest`, `icon-192.png`, `icon-512.png` — PWA-манифест и иконки.

## Внешние сервисы

- **Vercel** — хостинг продакшен-версии.
- **Supabase Auth** — вход сотрудников.
- **Supabase Database** — таблицы профилей, клиентов, замеров и фото.
- **Supabase Storage** — bucket `measurement-photos` для файлов фотографий.

## Роли

Текущая логика ролей завязана на профиль пользователя и клиентские проверки в приложении. Основные роли:

- `zamer` — замерщик, создаёт и заполняет замеры.
- `manager` / `admin` — менеджер или администратор, проверяет, принимает, архивирует, работает с корзиной.
- `constructor` / `production` / изготовитель — пользователь производственного режима, работает через экран «Для изготовителя».

RLS-политики Supabase должны соответствовать этим ролям. Клиентские проверки в JavaScript не заменяют серверные права.

## Offline-режим

Offline-режим состоит из двух частей:

1. **PWA/offline-start** — service worker кэширует основной app shell, чтобы приложение могло открыться без интернета после первого онлайн-открытия.
2. **TEMP-черновики** — данные замера и фото могут сохраняться локально в IndexedDB на телефоне, а затем синхронизироваться с Supabase.

Короткий сценарий проверки:

1. Открыть сайт онлайн.
2. Нажать «Обновить» 2–3 раза.
3. Убедиться, что вход сохранён.
4. Открыть «Технические действия» → «Проверить офлайн-доступ».
5. Должно появиться «Офлайн-режим готов».
6. Закрыть сайт.
7. Включить авиарежим.
8. Открыть приложение снова.
9. Создать TEMP-черновик.
10. Добавить фото.
11. Закрыть и открыть приложение — TEMP-черновик и фото должны остаться.

Подробнее: [docs/offline-mode.md](docs/offline-mode.md).

## Как обновлять приложение после merge

После merge PR в `main` и деплоя на Vercel:

1. Открыть приложение онлайн на телефоне.
2. Нажать «Обновить» 2–3 раза.
3. Закрыть и открыть приложение снова.
4. Проверить вход.
5. Проверить «Технические действия» → «Проверить офлайн-доступ».
6. Только после этого проверять авиарежим и TEMP-черновики.

Если менялись `app.js`, `service-worker.js`, `styles.css` или подключаемые скрипты, нужно особенно внимательно следить за cache-bust версиями и service worker cache version.

## Переменные окружения

В долгосрочной схеме Supabase-конфиг лучше вынести в env/build-step. Для этого добавлен пример:

```text
SUPABASE_URL=
SUPABASE_ANON_KEY=
```

Сейчас в проекте ключи могут быть прописаны напрямую в `app.js` / `production.js`. В этом документационном PR логика не меняется: мы только фиксируем текущее состояние и будущую цель.

## Checklist проверки после изменений

Минимальная проверка для маленького PR:

```bash
git diff --check
```

Если менялся JavaScript:

```bash
node --check app.js
node --check service-worker.js
```

Ручная проверка:

- открывается главный экран;
- работает вход;
- создаётся новый замер;
- сохраняется карточка клиента и замера;
- добавляется фото;
- открывается SVG-схема;
- работает режим «Для изготовителя»;
- работает корзина/восстановление;
- после онлайн-открытия приложение проходит проверку offline-доступа;
- в авиарежиме создаётся TEMP-черновик и сохраняется после перезапуска.
