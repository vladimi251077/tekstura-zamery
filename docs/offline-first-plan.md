# Offline-first режим замеров

## 1. Цель

Offline-first режим нужен, чтобы замерщик мог полноценно работать на объекте даже без стабильного интернета.

Основные цели:

- замерщик может работать на объекте без интернета;
- данные замера и фотографии сохраняются локально на телефоне;
- после появления интернета локальные данные синхронизируются с Supabase;
- текущая онлайн-логика приложения остаётся основной и не ломается;
- offline-first режим добавляется поэтапно и безопасно, без резких изменений бизнес-логики.

Ожидаемый результат: замерщик не теряет работу из-за плохой связи, а компания получает замер в Supabase после синхронизации.

## 2. Как это работает для замерщика

Пользовательский сценарий должен быть простым и предсказуемым.

1. Первый вход в приложение выполняется только онлайн.
   - Замерщик должен авторизоваться через текущий Auth-механизм.
   - Приложение должно загрузить необходимые файлы интерфейса и сохранить app shell в cache.

2. После первого успешного входа приложение может открываться без интернета.
   - Если браузер уже имеет сохранённую сессию, приложение показывает рабочий интерфейс.
   - Если сессии нет или она истекла, вход без интернета невозможен.

3. Без интернета создаётся TEMP-замер.
   - TEMP-замер получает локальный временный номер, например `TEMP-2026-05-17-001` или UUID.
   - Этот номер нужен только на устройстве до синхронизации.
   - После отправки в Supabase TEMP-замер получает реальный идентификатор/номер из основной онлайн-логики.

4. Размеры и данные замера сохраняются локально.
   - Все важные поля замера записываются в IndexedDB.
   - Автосохранение должно происходить после значимых изменений и перед закрытием вкладки.

5. Фото сохраняются локально на телефоне.
   - Каждое фото сохраняется как Blob/File в IndexedDB.
   - Для фото хранится связь с TEMP-замером.
   - Несколько фото должны поддерживаться сразу.

6. При появлении интернета замер отправляется в Supabase.
   - Пользователь видит, что есть несинхронизированные замеры.
   - Синхронизация может запускаться вручную на первых этапах.
   - В следующих этапах можно добавить автоматическую синхронизацию.

7. Production доступен только после синхронизации.
   - Для TEMP-замера production недоступен.
   - Production должен работать только с полноценным замером, который уже сохранён в Supabase.

## 3. Архитектура

Offline-first режим должен быть добавлен как отдельный слой вокруг текущей онлайн-логики, а не как её замена.

### `service-worker.js`

`service-worker.js` отвечает за базовую доступность приложения без интернета.

Задачи service worker:

- кэшировать offline app shell;
- отдавать сохранённые статические файлы при отсутствии сети;
- не вмешиваться в бизнес-логику сохранения замеров;
- не подменять Supabase API-ответы;
- не реализовывать синхронизацию на первом этапе.

На первом PR service worker должен быть максимально простым и безопасным.

### Offline app shell cache

App shell cache должен содержать только файлы, необходимые для открытия интерфейса приложения:

- HTML-точку входа;
- основные JS-файлы;
- CSS;
- критичные иконки/ресурсы, если они нужны для запуска интерфейса.

Важно:

- cache не должен ломать обновления приложения;
- нужна версия cache, например `app-shell-v1`;
- при новой версии старый cache должен очищаться;
- network-first или cache-first стратегия должна выбираться осторожно для каждого типа файлов.

Рекомендуемый подход:

- для статических файлов — cache-first с обновлением версии cache;
- для HTML — network-first с fallback на cache;
- для Supabase/API-запросов — не кэшировать как app shell.

### IndexedDB

IndexedDB используется как основное локальное хранилище offline-замеров.

Причины выбора IndexedDB:

- поддерживает большие объёмы данных лучше, чем localStorage;
- умеет хранить Blob/File;
- подходит для очередей синхронизации;
- работает в мобильных браузерах, включая Safari/iPhone, с ограничениями.

IndexedDB хранит:

- черновики замеров;
- фотографии;
- очередь синхронизации;
- статусы локальных операций.

### `offline-db.js`

`offline-db.js` должен быть helper-модулем для работы с IndexedDB.

Задачи:

- открыть базу и создать stores;
- добавить/обновить offline draft;
- получить список несинхронизированных замеров;
- сохранить и получить фото;
- добавить операции в sync queue;
- обновить статус операции;
- удалить локальные данные только после полной синхронизации.

На первом этапе helper может быть создан без подключения к реальной бизнес-логике.

### `offline-sync.js` в следующих этапах

`offline-sync.js` должен появиться позже, когда будет реализована реальная синхронизация.

Задачи:

- читать sync queue;
- последовательно отправлять данные в Supabase;
- создавать клиента, замер и связанные записи;
- загружать фото в Supabase Storage;
- создавать строки в `measurement_photos`;
- фиксировать успешные шаги;
- повторять неудачные шаги;
- не удалять локальные данные до полного успеха.

### Sync queue

Sync queue нужна, чтобы синхронизация была надёжной и повторяемой.

Очередь должна хранить операции, которые нужно выполнить онлайн:

- создать/обновить client;
- создать measurement;
- отправить JSON-данные проекта;
- отправить SVG;
- загрузить фото;
- создать строки фото;
- обновить статус замера.

Каждая операция должна иметь статус и количество попыток.

### Локальное хранение фото как Blob/File

Фото должны сохраняться локально в IndexedDB как Blob/File.

Для каждого фото нужно хранить:

- локальный идентификатор;
- идентификатор TEMP-замера;
- Blob/File;
- MIME-type;
- исходное имя файла, если есть;
- размер файла;
- дату создания;
- статус синхронизации;
- путь в Supabase Storage после успешной загрузки.

### Сжатие фото перед отправкой

Перед отправкой фото в Supabase Storage нужно сжимать.

Рекомендуемые параметры:

- длинная сторона: 1600–2000 px;
- качество JPEG/WebP: 0.75–0.85;
- сохранять достаточное качество для просмотра деталей объекта;
- избегать слишком больших файлов, особенно на мобильном интернете.

Сжатие можно делать:

- сразу при добавлении фото;
- или перед отправкой в Supabase.

Предпочтительно хранить локально уже сжатую версию, чтобы экономить место на телефоне.

## 4. IndexedDB stores

Предлагаемая база: `tekstura_offline`.

Версия базы должна увеличиваться при изменении структуры stores.

### `offline_drafts`

Store для локальных TEMP-замеров и черновиков.

Примерная структура записи:

```json
{
  "local_id": "temp_8f2b6d1e-1111-4444-9999-abcdef123456",
  "temp_number": "TEMP-2026-05-17-001",
  "status": "draft_offline",
  "sync_status": "pending",
  "created_at": "2026-05-17T10:00:00.000Z",
  "updated_at": "2026-05-17T10:15:00.000Z",
  "client": {
    "name": "Иван Иванов",
    "phone": "+79990000000",
    "address": "Адрес объекта"
  },
  "measurement": {
    "comment": "Комментарий замерщика",
    "object_type": "apartment",
    "local_measurement_number": "TEMP-2026-05-17-001"
  },
  "drawing_project_json": {},
  "finish_dimensions_json": {},
  "drawing_svg": "<svg></svg>",
  "photo_ids": [
    "photo_local_1",
    "photo_local_2"
  ],
  "supabase": {
    "client_id": null,
    "measurement_id": null
  },
  "last_error": null
}
```

Рекомендуемые индексы:

- `local_id` как keyPath;
- `sync_status`;
- `created_at`;
- `updated_at`.

Возможные `sync_status`:

- `local_only`;
- `pending`;
- `syncing`;
- `partial`;
- `synced`;
- `error`.

### `offline_photos`

Store для локальных фотографий.

Примерная структура записи:

```json
{
  "local_photo_id": "photo_local_1",
  "draft_local_id": "temp_8f2b6d1e-1111-4444-9999-abcdef123456",
  "blob": "Blob/File object",
  "mime_type": "image/jpeg",
  "original_name": "kitchen.jpg",
  "size": 345678,
  "width": 1800,
  "height": 1350,
  "created_at": "2026-05-17T10:05:00.000Z",
  "sync_status": "pending",
  "storage_bucket": null,
  "storage_path": null,
  "measurement_photo_id": null,
  "attempts": 0,
  "last_error": null
}
```

Рекомендуемые индексы:

- `local_photo_id` как keyPath;
- `draft_local_id`;
- `sync_status`;
- `created_at`.

Фото нельзя удалять из IndexedDB, пока:

- Blob успешно не загружен в Supabase Storage;
- строка в `measurement_photos` успешно не создана;
- весь замер не получил финальный статус синхронизации.

### `sync_queue`

Store для очереди операций синхронизации.

Примерная структура записи:

```json
{
  "queue_id": "queue_001",
  "draft_local_id": "temp_8f2b6d1e-1111-4444-9999-abcdef123456",
  "type": "create_measurement",
  "status": "pending",
  "payload": {},
  "depends_on": [
    "queue_000"
  ],
  "attempts": 0,
  "max_attempts": 5,
  "created_at": "2026-05-17T10:10:00.000Z",
  "updated_at": "2026-05-17T10:10:00.000Z",
  "last_error": null
}
```

Рекомендуемые индексы:

- `queue_id` как keyPath;
- `draft_local_id`;
- `status`;
- `type`;
- `created_at`.

Возможные `type`:

- `upsert_client`;
- `create_measurement`;
- `save_drawing_project_json`;
- `save_finish_dimensions_json`;
- `save_drawing_svg`;
- `upload_photo`;
- `create_measurement_photo`;
- `update_measurement_status`.

Возможные `status`:

- `pending`;
- `running`;
- `done`;
- `error`;
- `blocked`.

## 5. Синхронизация

Синхронизация должна быть последовательной, идемпотентной насколько возможно и устойчивой к обрывам сети.

Рекомендуемый порядок синхронизации одного TEMP-замера:

1. `client`
   - Найти или создать клиента по текущим правилам приложения.
   - Сохранить `client_id` в `offline_drafts.supabase.client_id`.
   - Если клиент уже создан при предыдущей попытке, не создавать дубль.

2. `measurement`
   - Создать основной замер в Supabase.
   - Передать `client_id` и основные поля замера.
   - Сохранить `measurement_id` в локальный draft.
   - TEMP-номер заменить на реальный идентификатор/номер, если это предусмотрено текущей логикой.

3. `drawing_project_json`
   - Отправить JSON проекта чертежа.
   - После успеха отметить соответствующую операцию очереди как `done`.

4. `finish_dimensions_json`
   - Отправить финальные размеры.
   - После успеха отметить операцию как `done`.

5. `drawing_svg`
   - Отправить или обновить SVG-чертёж.
   - SVG должен соответствовать уже созданному `measurement_id`.

6. `photos` в Supabase Storage
   - Для каждого локального фото выполнить загрузку в Storage.
   - Путь должен быть детерминированным, например `measurements/{measurement_id}/{local_photo_id}.jpg`.
   - После успешной загрузки сохранить `storage_path` в `offline_photos`.

7. Строки в `measurement_photos`
   - Для каждого загруженного фото создать строку в `measurement_photos`.
   - Строка должна ссылаться на `measurement_id` и `storage_path`.
   - После успеха сохранить `measurement_photo_id` локально.

8. Статус замера
   - После успешной отправки всех данных и фото обновить статус замера.
   - Локальный draft получает `sync_status: synced`.
   - Только после этого локальные данные можно пометить как безопасные для очистки.

Если интернет оборвался на любом шаге:

- уже успешные шаги не повторяются вслепую;
- незавершённые операции остаются в `sync_queue`;
- пользователь видит ошибку или статус частичной синхронизации;
- следующая попытка продолжает процесс с последнего незавершённого шага.

## 6. Фото

Фото — самая рискованная часть offline-first режима, потому что они занимают много места и часто отправляются по нестабильному мобильному интернету.

### Сохранение фото локально

При добавлении фото без интернета приложение должно:

- получить File из камеры или галереи;
- при необходимости прочитать метаданные изображения;
- сжать изображение до допустимого размера;
- сохранить Blob/File в `offline_photos`;
- добавить `local_photo_id` в `offline_drafts.photo_ids`;
- показать фото в интерфейсе как локально сохранённое.

### Несколько фото

Один TEMP-замер должен поддерживать несколько фото.

Для каждого фото нужен отдельный статус:

- локально сохранено;
- ожидает отправки;
- отправляется;
- отправлено в Storage;
- запись `measurement_photos` создана;
- ошибка отправки.

Ошибка одного фото не должна удалять остальные фото и не должна ломать весь локальный draft.

### Сжатие

Рекомендуемые параметры сжатия:

- длинная сторона изображения: 1600–2000 px;
- качество: 0.75–0.85;
- формат: JPEG или WebP, если WebP безопасно поддерживается текущими целевыми браузерами;
- сохранять ориентацию изображения;
- избегать потери важных деталей.

Если исходное фото уже меньше целевого размера, его можно не масштабировать, но всё равно можно применить умеренное JPEG-сжатие.

### Повторная отправка

Если интернет оборвался во время отправки фото:

- локальный Blob/File остаётся в IndexedDB;
- операция `upload_photo` остаётся в `sync_queue` со статусом `pending` или `error`;
- количество попыток увеличивается;
- при следующей синхронизации отправка повторяется;
- если Storage upload уже прошёл, но строка `measurement_photos` не создана, повторяется только создание строки.

## 7. UI

Пользователь должен всегда понимать, что происходит с сетью и локальными данными.

Рекомендуемые статусы UI:

### Онлайн

Показывается, когда `navigator.onLine === true` и приложение может выполнять сетевые операции.

Текст для пользователя:

- `Онлайн`;
- `Интернет подключён`.

### Офлайн — данные сохраняются в телефоне

Показывается, когда сеть недоступна.

Текст для пользователя:

- `Офлайн — данные сохраняются в телефоне`;
- `Интернета нет. Замер сохранится локально и отправится позже`.

### Есть несинхронизированные замеры

Показывается, если в IndexedDB есть drafts или queue operations со статусом `pending`, `partial` или `error`.

Текст для пользователя:

- `Есть несинхронизированные замеры`;
- `Нужно синхронизировать данные при подключении к интернету`.

### Синхронизация

Показывается во время отправки данных.

Текст для пользователя:

- `Синхронизация...`;
- `Отправляем замер и фото в Supabase`.

Желательно показывать прогресс:

- данные замера;
- фото 1 из N;
- завершение.

### Синхронизировано

Показывается после полного успеха.

Текст для пользователя:

- `Синхронизировано`;
- `Замер отправлен. Production доступен`.

### Ошибка синхронизации

Показывается, если часть операций завершилась ошибкой.

Текст для пользователя:

- `Ошибка синхронизации`;
- `Данные сохранены в телефоне. Повторите синхронизацию позже`.

Важно: ошибка не должна пугать пользователя потерей данных, если данные действительно остались в IndexedDB.

## 8. Ограничения

Offline-first режим имеет важные ограничения, которые нужно явно учитывать в разработке и UX.

- Первый вход без интернета невозможен.
  - Пользователь должен хотя бы один раз войти онлайн.
  - Без сохранённой авторизованной сессии приложение не сможет безопасно работать.

- Production для TEMP-замера недоступен.
  - TEMP-замер существует только локально.
  - Production должен открываться только после создания реального замера в Supabase.

- IndexedDB на iPhone может быть очищен системой.
  - iOS Safari может удалить данные сайта при нехватке места или длительном неиспользовании.
  - Нужно предупреждать пользователя, что несинхронизированные замеры нельзя долго хранить только на телефоне.

- Нельзя удалять локальный замер до полной синхронизации.
  - Локальные данные можно чистить только после успешной отправки client, measurement, JSON, SVG, всех фото и строк `measurement_photos`.
  - При частичной синхронизации локальные данные должны оставаться.

- Нельзя менять Supabase schema без отдельного согласования.
  - Если для offline-first режима потребуются новые поля или таблицы, это должно идти отдельным согласованным этапом.

- Нельзя ломать текущую онлайн-логику.
  - Existing online save flow должен продолжать работать как раньше.
  - Offline-first слой должен добавляться постепенно.

## 9. Этапы PR

Offline-first режим нужно внедрять маленькими PR, чтобы снизить риск поломок.

### PR 1

Цель: подготовить техническую основу без реальной синхронизации и без изменения бизнес-логики.

Состав:

- только `service-worker.js`;
- registration service worker;
- online/offline indicator;
- `offline-db.js` helper;
- базовая структура IndexedDB;
- без реальной синхронизации;
- без изменения бизнес-логики;
- без изменения текущего онлайн-сохранения.

Проверки PR 1:

- приложение открывается онлайн как раньше;
- после первого открытия app shell доступен офлайн;
- индикатор сети меняется при offline/online;
- текущие онлайн-замеры сохраняются без изменений.

### PR 2

Цель: добавить локальные offline drafts без фото.

Состав:

- локальные offline drafts без фото;
- TEMP-номера;
- автосохранение в IndexedDB;
- восстановление draft после закрытия вкладки;
- ручная синхронизация данных без фото;
- production заблокирован для TEMP-замеров.

## 10. Что нельзя трогать

В рамках подготовки offline-first режима нельзя трогать без отдельного согласования:

- Auth/логины;
- SVG-расчёты;
- production layout;
- сапожки;
- размеры площадки;
- текущую онлайн-логику сохранения;
- Supabase schema без отдельного согласования.

Также нельзя:

- переписывать существующие сценарии сохранения;
- менять структуру production без отдельного PR;
- создавать миграции Supabase без согласования;
- смешивать документационный PR с реализацией.

## 11. Проверки

### Ручные тесты

Полный ручной сценарий для будущих PR с реализацией:

1. Войти онлайн.
2. Убедиться, что приложение загрузилось и сессия активна.
3. Включить авиарежим.
4. Открыть приложение.
5. Создать замер.
6. Заполнить основные данные и размеры.
7. Добавить фото.
8. Закрыть вкладку.
9. Открыть приложение снова без интернета.
10. Проверить, что замер и фото сохранились.
11. Включить интернет.
12. Запустить синхронизацию.
13. Проверить, что данные ушли в Supabase.
14. Проверить, что фото появились в Supabase Storage.
15. Проверить строки `measurement_photos`.
16. Проверить, что production стал доступен только после синхронизации.
17. Проверить, что TEMP-замер получил реальный идентификатор/номер.

### Команды проверки

Для каждого PR с кодом нужно запускать:

```bash
node --check app.js
node --check drawing-bridge.js
node --check production.js
node --check details-enhance.js
node --check photo-preview.js
git diff --check
```

Для документационного PR достаточно проверить:

```bash
git diff --check
```
