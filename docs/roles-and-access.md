# Роли и доступы

Роли нужны, чтобы разные сотрудники видели только нужные действия. Клиентские проверки в JavaScript помогают интерфейсу, но настоящая защита должна быть в Supabase RLS.

## Основные роли

### `zamer`

Замерщик.

Типовые действия:

- войти в приложение;
- создать новый замер;
- заполнить клиента, размеры, комментарии;
- добавить фото;
- сохранить SVG-схему;
- создать TEMP-черновик без интернета;
- синхронизировать свой TEMP-черновик после появления интернета;
- отправить замер на проверку, если данные заполнены.

### `manager` / `admin`

Менеджер или администратор.

Типовые действия:

- видеть рабочие замеры;
- проверять карточки;
- менять статусы;
- принимать замер;
- переводить в архив;
- работать с корзиной;
- восстанавливать замеры;
- выполнять окончательное удаление из Supabase при наличии служебного кода.

`admin` может иметь расширенные права, но это должно подтверждаться RLS-политиками и текущей схемой Supabase.

### `constructor` / `production` / изготовитель

Изготовитель или конструктор.

Типовые действия:

- открыть режим «Для изготовителя»;
- видеть подготовленные размеры, схему и фото;
- работать с производственными статусами, если роль это разрешает.

В коде производственный экран также учитывает роли вроде `production`, `prod`, `цех`, `производ`, `montage`, `монтаж`.

## Кто может создавать замеры

Создание замера должно быть доступно замерщику и ролям, которым разрешено редактирование. Минимальная роль для выездной работы — `zamer`.

Серверная политика должна проверять `auth.uid()` и роль из `profiles`.

## Кто может принимать и архивировать

Принятие, перевод в архив и производственные действия должны быть доступны менеджеру/администратору и тем ролям, которым это явно разрешено.

Перед принятием приложение проверяет обязательные поля: клиентские данные, размеры и сохранённую схему/SVG. Эти проверки помогают не отправить пустой замер, но не заменяют серверные правила.

## Корзина

Корзина работает как soft delete.

Замер не удаляется сразу из Supabase, а получает признаки удаления, например:

- `is_deleted`;
- `deleted_at`;
- `deleted_by`.

Такой замер можно восстановить.

Окончательное удаление доступно только для замеров из корзины и дополнительно защищено служебным кодом, указанным в `PERMANENT_DELETE_PASSWORD` в `app.js`.

## Окончательное удаление

При окончательном удалении должны удаляться:

1. файлы фото из Storage bucket `measurement-photos`;
2. строки из `measurement_photos`;
3. строка замера из `measurements`;
4. клиент из `clients`, если у него больше нет других замеров.

Это действие нельзя отменить.

## Какие RLS-политики могут потребоваться

Минимально нужно проверить политики для:

- `profiles`;
- `clients`;
- `measurements`;
- `measurement_photos`;
- `storage.objects` для bucket `measurement-photos`.

Примерные правила на уровне смысла:

- пользователь может читать свой профиль;
- замерщик может создавать и редактировать свои замеры;
- менеджер/admin может читать и менять все рабочие замеры;
- производственная роль может читать замеры, готовые для производства;
- удаление и очистка корзины разрешены только manager/admin;
- `measurement_photos` должны быть доступны только в пределах доступных замеров;
- Storage objects должны разрешать upload/read/delete в соответствии с правами на связанные замеры.

Точную SQL-реализацию RLS нужно делать отдельным PR/миграцией и проверять на тестовых данных.
