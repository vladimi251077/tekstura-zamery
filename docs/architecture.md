# Архитектура Tekstura Замеры

Этот документ описывает текущую архитектуру проекта как есть. Он не описывает идеальную архитектуру и не предлагает переписать приложение сразу.

Проект уже работает в продакшене, поэтому основное правило разработки: маленькие безопасные PR, минимум одновременных изменений и обязательная ручная проверка на телефоне.

## Общая схема

Tekstura Замеры — статическое PWA-приложение на HTML, CSS и JavaScript. Основная логика находится в клиенте. Данные хранятся в Supabase, а offline-данные временно хранятся на устройстве пользователя через IndexedDB.

Основной поток:

1. Пользователь открывает `index.html`.
2. Подключаются стили, Supabase client, `offline-db.js`, `app.js` и вспомогательные скрипты.
3. `app.js` инициализирует авторизацию, профиль, список замеров и UI.
4. Замеры, клиенты и фото читаются/пишутся в Supabase.
5. Service worker кэширует app shell для offline-start.
6. TEMP-черновики и offline-фото сохраняются в IndexedDB и синхронизируются позже.

## `index.html`

`index.html` — главный экран приложения. В нём находятся:

- PWA meta-теги;
- подключение `manifest.webmanifest`;
- подключение Supabase SDK;
- экран входа;
- основной рабочий экран;
- список замеров;
- форма замера;
- вкладки «Общее», «Размеры», «Фото», «Проверка»;
- блок «Технические действия»;
- ссылка на режим «Для изготовителя»;
- кнопка проверки offline-доступа.

Файл не должен превращаться в место для новой бизнес-логики. Новые действия лучше добавлять маленькими изменениями и постепенно выносить в отдельные модули.

## `app.js`

`app.js` — основной runtime приложения. Сейчас это самый большой и самый важный файл проекта.

В нём находятся:

- Supabase-конфиг и создание клиента;
- состояние приложения;
- авторизация и запоминание входа на телефоне;
- загрузка профиля;
- роли и UI-доступы;
- создание и редактирование замеров;
- работа с клиентами;
- статусы замеров;
- режимы простого/детального замера;
- SVG/чертёжные поля;
- фото;
- online-сохранение в Supabase;
- offline TEMP-черновики;
- синхронизация TEMP-черновиков;
- синхронизация offline-фото;
- корзина, восстановление и полное удаление;
- проверки перед отправкой/принятием замера.

Пока нельзя делать большой рефакторинг `app.js` одним PR. Любой вынос логики должен идти по одному направлению за раз и сопровождаться ручной проверкой.

## `offline-db.js`

`offline-db.js` — отдельный слой для IndexedDB.

Текущая база:

- database: `tekstura-offline-shell`;
- stores:
  - `metadata`;
  - `offline_drafts`;
  - `sync_queue`;
  - `offline_photos`.

Здесь хранятся TEMP-черновики, offline-фото, статусы синхронизации и служебные метаданные. Этот файл должен оставаться стабильным, потому что данные уже могут лежать на телефонах пользователей.

## `service-worker.js`

`service-worker.js` отвечает за PWA/offline-start.

Он:

- создаёт app shell cache;
- кэширует главную страницу, стили, основные JS-файлы, manifest и иконки;
- использует network-first для HTML/JS/CSS;
- использует cache-first для manifest и изображений;
- не обрабатывает Supabase-запросы;
- исключает `production.html`, `production.js`, `production.css` и `svg-constructor/` из локального app shell cache;
- при отсутствии shell показывает понятный fallback.

Важно: при изменении app shell файлов нужно внимательно обновлять cache-bust версии и cache version.

## `drawing-bridge.js`

`drawing-bridge.js` связывает форму замера с SVG-схемой. Он нужен, чтобы карточка замера могла получать/сохранять данные схемы и SVG без полного переписывания основного приложения.

Этот слой нужно развивать осторожно, потому что SVG-схема используется в основном интерфейсе и в производственном просмотре.

## `production.html` / `production.js`

`production.html` и `production.js` — отдельный режим «Для изготовителя».

Задача режима:

- показать замеры, готовые для производства;
- дать производству/конструктору смотреть размеры, схему и фото;
- работать без лишнего интерфейса замерщика;
- позволять менять производственные статусы тем ролям, которым это разрешено.

Этот режим не входит в основной PWA shell service worker. Его нельзя случайно сломать при изменениях основного приложения.

## Supabase Database

Текущая логика использует Supabase Database как основной источник данных.

Основные таблицы, которые видны по коду:

- `profiles` — профиль пользователя и роль;
- `clients` — клиент, телефон, адрес;
- `measurements` — карточка замера, размеры, статусы, SVG, JSON схемы;
- `measurement_photos` — записи о фотографиях, связанных с замером.

Для удаления и корзины используются поля замеров вроде `is_deleted`, `deleted_at`, `deleted_by`, `is_archived` и статусы.

Точные RLS-политики должны храниться и проверяться в Supabase. Клиентская проверка роли в `app.js` не является защитой базы.

## Supabase Storage

Фото хранятся в Supabase Storage bucket:

```text
measurement-photos
```

В таблице `measurement_photos` хранится путь к файлу и метаданные. Для просмотра приложение получает signed URL.

## Почему пока не делаем большой рефакторинг

Причины:

- приложение уже используется в продакшене;
- offline/PWA-логика чувствительна к cache version и IndexedDB;
- на телефонах могут быть локальные TEMP-черновики;
- фото и синхронизация завязаны на Supabase Storage и `measurement_photos`;
- один большой PR сложнее проверить;
- ошибка в одном месте может сломать выездные замеры.

Правильный подход — сначала документация, затем маленькие PR по одному модулю: фото, offline sync, auth, Supabase API, расчёты, SVG.
