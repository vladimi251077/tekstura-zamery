# План безопасного рефакторинга

Цель рефакторинга — сделать проект проще для поддержки, но не сломать рабочее приложение замерщиков.

Главное правило: не переписывать весь проект сразу.

## Почему нельзя делать большой рефакторинг одним PR

- приложение уже работает в продакшене;
- замерщики могут хранить TEMP-черновики на телефонах;
- offline/PWA зависит от service worker cache version;
- фото завязаны на Storage и `measurement_photos`;
- `app.js` содержит много связанной бизнес-логики;
- большой PR сложно проверить на телефоне;
- ошибка может остановить реальные замеры.

## Безопасная последовательность

### 1. Документация

Сначала фиксируем текущее состояние:

- README;
- архитектура;
- offline-mode;
- фото;
- роли;
- план рефакторинга.

Это даёт опору перед любыми изменениями кода.

### 2. `photos.js` / photo helpers

Первый кандидат на вынос — фото.

Можно постепенно вынести:

- safe file name / extension helpers;
- upload single photo;
- delete photo;
- signed URL helper;
- offline photo helpers;
- render-independent функции, если они не завязаны на DOM.

Важно: сначала выносить чистые функции, потом функции с Supabase, и только потом UI.

### 3. `offline-sync.js`

После фото можно отделить синхронизацию TEMP-черновиков:

- build payload from offline draft;
- mark syncing/synced/error;
- sync draft;
- sync offline photos;
- retry logic.

Это чувствительная часть. Нужны ручные тесты на телефоне.

### 4. `auth.js`

Дальше можно отделить авторизацию:

- login normalization;
- remembered auth;
- read stored Supabase session;
- profile loading;
- role helpers.

В этом PR нельзя ломать сохранённый вход на телефоне.

### 5. `supabase-api.js`

После стабилизации auth и sync можно сделать тонкий слой API:

- clients API;
- measurements API;
- photos API;
- profile API;
- trash/delete API.

Цель — чтобы `app.js` меньше знал о `.from(...)`, `.insert(...)`, `.update(...)`.

### 6. Measurements calculations

Чистые расчёты размеров лучше выносить отдельно:

- required fields;
- tread mode;
- march length;
- finish dimensions;
- проверки перед принятием.

Это лучший участок для первых unit-тестов.

### 7. Drawing / SVG modules

SVG и drawing-bridge выносить позже, когда остальные части станут стабильнее.

Причина: SVG связан с UI, сохранением `drawing_project_json`, `drawing_svg`, режимом изготовителя и проверками размеров.

## TypeScript

TypeScript пока отложить.

Причина: миграция на TypeScript потребует build-step, типизацию Supabase, настройку деплоя и может затронуть много файлов сразу.

Лучше сначала уменьшить `app.js` маленькими JS-модулями.

## Zustand / Valtio

Zustand, Valtio или другое state-management решение пока не внедрять.

Причина: текущее состояние приложения завязано на DOM и прямые функции. Новое состояние поверх старого может создать два источника правды.

Сначала нужно выделить модули и чистые функции.

## Тесты

Начинать тесты лучше не с UI, а с чистых функций:

- расчёт обязательных полей;
- расчёт длины маршей;
- tread mode;
- finish dimensions;
- нормализация offline draft;
- подготовка payload для Supabase;
- фильтрация фото текущего замера.

После этого можно добавить минимальные интеграционные проверки для offline sync.

## Правило для каждого будущего PR

Каждый PR должен отвечать на вопросы:

- какие файлы изменены;
- затронута ли логика PWA/offline;
- затронуты ли фото;
- затронут ли Supabase schema/RLS;
- нужен ли cache-bust;
- как проверить на телефоне;
- что может сломаться;
- как быстро откатить.
