# Offline-режим и PWA

Offline-режим нужен, чтобы замерщик мог открыть приложение и создать TEMP-черновик на объекте даже без интернета.

Он состоит из двух разных механизмов:

1. **PWA/offline-start** — приложение открывается без интернета, потому что service worker заранее сохранил app shell.
2. **TEMP-черновики** — данные замера и фото сохраняются локально в IndexedDB на телефоне и отправляются в Supabase позже.

## Как работает PWA/offline-start

После первого открытия сайта онлайн браузер регистрирует `service-worker.js`.

Service worker создаёт app shell cache и сохраняет основные файлы:

- `./`;
- `./index.html`;
- `styles.css`;
- `app.js`;
- `offline-db.js`;
- `photo-preview.js`;
- `details-enhance.js`;
- `drawing-bridge.js`;
- `manifest.webmanifest`;
- иконки PWA.

Для HTML, JS и CSS используется network-first: если интернет есть, берётся свежая версия и кладётся в cache; если интернета нет, берётся cached версия.

Если app shell ещё не подготовлен, service worker должен показать понятное сообщение, что нужно открыть сайт один раз при интернете и нажать «Обновить».

## Что кэширует service worker

Service worker кэширует только локальный app shell. Он не кэширует Supabase API и Storage-запросы.

Исключения из app shell cache:

- `production.html`;
- `production.js`;
- `production.css`;
- `svg-constructor/`;
- любые Supabase-запросы.

Это важно: offline-start нужен для открытия основного приложения, а не для полной offline-копии всей базы.

## TEMP-черновики

TEMP-черновик — локальный замер, который ещё не создан в Supabase.

Он хранится только на конкретном телефоне в IndexedDB. До синхронизации у него нет нормального серверного `measurement.id`.

В TEMP-черновике сохраняются:

- данные клиента;
- данные замера;
- `drawing_project_json`;
- `finish_dimensions_json`;
- `drawing_svg`;
- имя/логин замерщика;
- статус синхронизации;
- ошибки синхронизации, если были.

TEMP-черновик можно создать без интернета, заполнить и позже отправить в Supabase.

## Где хранятся offline-фото

Offline-фото хранятся в IndexedDB store `offline_photos`.

Для фото сохраняются:

- локальный id фото;
- локальный id TEMP-черновика;
- blob файла;
- имя файла;
- MIME type;
- размер;
- тип фото;
- статус синхронизации;
- путь в Storage после успешной отправки;
- id строки `measurement_photos` после успешной отправки.

Перед сохранением приложение пытается сжать фото в JPEG, чтобы уменьшить вес локального хранилища и будущей отправки.

## Синхронизация

Когда интернет появился, TEMP-черновик можно синхронизировать.

Порядок:

1. Создаётся/обновляется клиент в `clients`.
2. Создаётся замер в `measurements`.
3. Локальный TEMP получает `server_id` и `server_number`.
4. Offline-фото отправляются в Storage bucket `measurement-photos`.
5. Для каждого фото создаётся строка в `measurement_photos`.
6. Локальные записи фото получают статус `synced`.

Если часть фото не отправилась, локальные фото остаются на телефоне. Пользователь может попробовать синхронизацию ещё раз.

## Что делать, если нет интернета

Если приложение уже было подготовлено для offline-start:

1. Открыть приложение с иконки или из браузера.
2. Создать локальный TEMP-черновик.
3. Заполнить данные и схему.
4. Добавить фото.
5. Закрыть приложение можно — данные должны остаться на телефоне.
6. Когда интернет появится, открыть TEMP-черновик и синхронизировать.

Если приложение не было подготовлено, offline-start может не сработать. Тогда нужно открыть сайт онлайн и подготовить app shell.

## Проверка offline-доступа

Ручной сценарий:

1. Открыть сайт онлайн.
2. Нажать «Обновить» 2–3 раза.
3. Убедиться, что пользователь вошёл.
4. Открыть «Технические действия».
5. Нажать «Проверить офлайн-доступ».
6. Ожидаемый результат: «Офлайн-режим готов».
7. Закрыть сайт.
8. Включить авиарежим.
9. Открыть приложение снова.
10. Создать TEMP-черновик.
11. Добавить фото.
12. Закрыть и открыть приложение.
13. Убедиться, что TEMP-черновик и фото сохранились.

## Если приложение не открывается в авиарежиме

Проверить:

1. Был ли сайт открыт онлайн после последнего merge/deploy.
2. Нажимали ли «Обновить» 2–3 раза.
3. Показывает ли healthcheck «Офлайн-режим готов».
4. Не менялись ли `app.js`, `styles.css`, `service-worker.js` без обновления cache-bust/cache version.
5. Не заблокирован ли service worker браузером.
6. Открывается ли приложение через тот же домен, где оно было подготовлено.
7. Не очищались ли данные сайта в браузере.
8. На iPhone — было ли приложение добавлено на экран «Домой» после актуального деплоя.

Если service worker не подготовил shell, нужно открыть приложение онлайн и повторить проверку.
